迁移到Swift 3

最近将公司项目从Swift 2.3迁移到Swift 3.0，总共用了四个工作日多点，一个人做的迁移和code review。主要是不能完全依赖于Xcode的refractor，哪怕是refracted出来的代码还是需要再次去review，否则很有可能影响原来的逻辑和功能，所以花的时间还是比较多的。

准备工作

  1. 知识准备

    在迁移之前我阅读了一篇Swift官方的Migration Guide, 这里基本上涵盖了大部分迁移时可能遇到的问题。

    其次我还阅读了一篇Airbnb迁移到Swift 3的实践，这里附上汉化版本的。

  2. 收拾心情，做好长时间专注的准备

开始

第三方依赖库的升级

由于我们项目使用的是cocoapods，所以需要先去pod search下最新版本的库，看看有没有支持到Swift 3.0。如果某个库没有支持到Swift 3.0，那么就只能做成私有库。我们项目中有个播放器的库还没有支持Swift 3.0的版本，所以我们就只能对Swift 2.3的代码进行迁移并做成私有库。这部分的逻辑比较简单，直接用Xcode的refractor迁移就可以了，最后别忘记将Build Setting->Use Legacy Swift Language Version改成No。至于如何做cocoapods的私有库，大家可以去网上搜索相关资料。

本身项目的升级

下面罗列的不仅仅是Swift 3.0的改变，也包含了我们项目中存在的不严谨，通过这次迁移得到的改善。

  1. Swift 3.0中的方法、枚举值命名规范不同于Swift 2.3中的

    上面是UITableViewCell被选中时调用的方法，明显可以看出first argument label被省略了。我们如果去调用一个含有underscore的function时，我们可以省略那个参数的argument label。但是在Swift 2.3中默认就是省略第一个参数的argument label，即使你显示声明了argument label。

    下面是官方文档的详解：

    Omitting Argument Labels
    If you don’t want an argument label for a parameter, write an underscore (_) instead of an explicit argument label for that parameter.

    If a parameter has an argument label, the argument must be labeled when you call the function.

    The Swift Programming Language (Swift 3)

    而枚举值的变更只是从uppercase统一转换为lowercase。

  1. Swift 3.0环境中编译器会检测closure的escaping或者nonescaping状态

    在Swift 2.3时候编译器对上述方法的声明没有任何警告，但是在Swift 3.0下，编译器有了警告。为什么需要在 (\ success: Bool) -> Void_ 这个closure前加@escaping修饰符呢？其实这是有上下文的，因为在我们项目中，给这个actionBlock赋值的时候，是传的一个函数。

    因为在watchContact function中有网络请求，并且completion closure是在请求完成后才调用的，而watchContact方法在这之前就已经return了，也就是这个closure逃逸了。(备注：watchContact中的@escaping也是这次升级才加上去的)

    所以Swift 3中编译器对闭包的escaping或者nonescaping的状态检测更加完善，但是一个closure是不是需要加上escaping修饰符，应该是我们开发者需要去思考的事，不能依赖于编译器去发现这种疏漏。

    下面是官方文档的详解：

    Escaping Closures
    A closure is said to escape a function when the closure is passed as an argument to the function, but is called after the function returns. When you declare a function that takes a closure as one of its parameters, you can write @escaping before the parameter’s type to indicate that the closure is allowed to escape.

    One way that a closure can escape is by being stored in a variable that is defined outside the function. As an example, many functions that start an asynchronous operation take a closure argument as a completion handler. The function returns after it starts the operation, but the closure isn’t called until the operation is completed—the closure needs to escape, to be called later.

    The Swift Programming Language (Swift 3)
